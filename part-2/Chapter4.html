What is CI/CD?

Let's break down the terms:

Continuous Integration (CI):

  • Concept: The practice of developers frequently merging their code changes (often daily or multiple times a day) into a central repository (like our main branch in GitLab).

  • Automation: Each merge triggers an automated process that builds the code and runs automated tests (unit tests, integration tests, linting checks, etc.).

  • Goal: To detect integration errors, bugs, and inconsistencies early and often, making them easier and faster to fix. It ensures the code in the main branch is always in a potentially shippable state.

Continuous Delivery (CD):

  • Concept: Extends CI by automating the release process. After the CI phase (build and test) passes successfully, the code changes are automatically prepared for release to a production-like environment (e.g., staging).

  • Automation: The process includes building release artifacts and potentially deploying them automatically to a staging environment.

  • Goal: To ensure that any code passing the automated checks can be released to users quickly and reliably with minimal manual intervention. The actual deployment to production might still require a manual approval step.

Continuous Deployment (CD):

  • Concept: Goes one step further than Continuous Delivery. Every change that passes all stages of the automated pipeline (build, test, staging deployment checks) is automatically deployed to production.

  • Automation: Fully automates the entire release process, including production deployment.

  • Goal: To release new features and fixes to users as quickly and frequently as possible, provided the automated checks are robust enough to ensure stability.

Why CI/CD for Infrastructure as Code (IaC)?

"Applying CI/CD principles to infrastructure defined with tools like CDK offers massive benefits," Maya emphasized.

  • Consistency: Automated pipelines execute the same steps every time, eliminating variations caused by different developer environments or manual errors during deployment (cdk deploy).

  • Speed & Frequency: Automating the testing and deployment process allows for faster feedback cycles and more frequent, smaller infrastructure updates, reducing the risk associated with large, infrequent changes.

  • Safety & Risk Reduction: Automated checks (linting, cdk diff, policy checks, security scans) can be built into the pipeline to catch potential issues before infrastructure is modified. Merge Request pipelines provide a crucial validation gate.

  • Reduced Manual Toil: Automates repetitive tasks like installing dependencies, compiling code, synthesizing templates, and running deployment commands, freeing up engineers for more valuable work.

  • Audit Trail & Visibility: CI/CD platforms like GitLab provide a clear history of what changes were deployed, when, by whom (or what process), and whether the associated checks passed or failed.

  • Collaboration: Provides a shared, automated process for the entire team, ensuring everyone follows the same deployment standards.

"Essentially, CI/CD brings the same rigor and automation benefits enjoyed in application development to the world of infrastructure management."

GitLab CI/CD Overview

"GitLab has a powerful and tightly integrated CI/CD system built right in," Maya said, bringing up a conceptual diagram. "The core components work together like this:"

Diagram: Conceptual flow of GitLab CI/CD.

"Let's define these terms:

  1. .gitlab-ci.yml file: This is a YAML file you add to the root directory of your repository. It defines the structure and order of your CI/CD pipeline: the stages, the jobs within those stages, and the scripts each job should run. This file is version-controlled along with your code.

  2. Pipeline: Represents one run of your CI/CD process, typically triggered by an event like a push to a branch or a merge request creation. A pipeline contains one or more stages.

  3. Stage: A logical grouping of jobs. Stages run sequentially (e.g., the test stage only runs after all jobs in the build stage succeed).

  4. Job: Defines a specific task to be executed, containing the actual shell scripts to run (e.g., npm install, npx cdk synth, npx cdk deploy). Jobs within the same stage typically run in parallel if enough Runners are available.

  5. GitLab Runner: An agent (a separate process or machine) that executes the jobs defined in your .gitlab-ci.yml file. You can use shared Runners provided by GitLab.com, or you can install and register your own self-hosted Runners (e.g., on EC2 or in Docker) which connect to your GitLab instance (either GitLab.com or your self-hosted server)."

Key Takeaways

  • Continuous Integration (CI) focuses on frequent code integration, automated builds, and testing to catch issues early.

  • Continuous Delivery/Deployment (CD) extends CI to automate the release and deployment process, improving speed and reliability.

  • CI/CD is crucial for IaC to ensure consistent, safe, and efficient infrastructure updates, reducing manual errors and toil.

  • GitLab CI/CD relies on a .gitlab-ci.yml file to define pipelines.

  • Pipelines consist of sequential Stages.

  • Stages contain parallel Jobs that execute specific scripts.

  • GitLab Runners are agents that execute the jobs.
